## 确认的技术方案

- 聚宽正式回测和模拟交易只在云端运行；本地只归档、查询和复盘。
- 历史运行必须由用户或 Agent（代理）明确指定；活动模拟交易每天北京时间 04:00 增量同步。
- 生产浏览器使用项目已有 Playwright（浏览器自动化）和仓库外专用登录目录；Codex 浏览器不承担计划任务。
- 每种预期数据都必须有独立完整性状态，不能用部分完整替代整体完整。
- 原始证据与 Parquet（列式数据）查询层共用一个页面对象清单和完整性门禁。
- OpenSpec（开放规格）是需求事实源；下方列出的边界条件将在 Design Doc（设计文档）阶段作为 Spec Patch 回写。
- 已确认架构：一个 Python CLI（命令行入口）作为唯一同步核心；Browser Adapter（浏览器适配器）、Research Exporter（研究数据导出器）、Ingest and Verify（导入与校验）、Archive and Query（归档与查询）四个边界清晰的组成部分。
- 已确认数据流：明确目标 → 前置清单 → 浏览器证据与 Research 数据 → 临时证据包 → 完整性校验 → 后置清单 → 原子落盘 → 查询/CSV。
- 已确认失败批次不能覆盖上一次完整归档，大文件不经过 Agent 对话，Cookie（浏览器凭证）不导出。
- 已确认身份与增量模型：策略和模拟交易使用不可变本地编号，回测/构建使用页面内稳定序号；远端标识只作别名；模拟交易使用不可变日期/游标分片，最后原子替换 manifest（清单）。
- 已确认对象门禁：必需核心数据全部 `complete` 且其他数据有明确证据状态时 `gate: pass`；报告必须保留 `capped_free` 等例外，不能把对象通过表述为所有日志完整。
- 已确认普通日志规则：默认必须拉取免费接口可得内容并至少覆盖前 1000 条；少于 1000 条且明确结束时为 `complete`。到达 1000 条后必须执行边界探测；只有确认后续存在但免费接口不可取，或无法证明结束时，才标记 `capped_free`。
- 已确认归因日志是独立核心数据集，不归入普通日志抽样或 1000 条边界；源站存在归因写入器时必须完整下载，并独立校验 Token、序号、起止事件、记录数和摘要。归因日志不完整时对象门禁失败；源站确实没有写入器时只能标记 `missing_at_source`，不得伪造。
- 已确认日志与异常管道：所有响应先保存原始压缩证据再解析；归因日志按 Token、序号、起止事件校验；积分日志必须按运行/类型/范围预览价格并单独确认；网络失败可重试，认证失效立即 `auth_required`，完整性失败不覆盖正式归档。
- 已确认同步结束时重读远端清单；清单漂移只重试变化部分，仍不一致时不提交该批次。
- 已确认取消 Plugin/marketplace（插件/市场）层：Codex 和 Claude 共用仓库内单一 Skill（技能），同步核心、CLI（命令行入口）、依赖与参考资料全部自包含在 Skill 目录；Windows Task Scheduler（Windows 任务计划程序）调用同一 Skill 脚本。
- Plugin 方案因双市场、双 manifest、安装缓存、版本同步和宿主重启要求被否决；`D:/My Project/my-agent-skills` 的双宿主结构只作为可行性参考，不复制其发布框架。
- 官方 Codex 文档确认仓库 Skill 使用 `.agents/skills/<name>/`，支持自包含 `scripts/`、自动检测变更和符号链接；官方建议单仓库工作流优先直接使用 Skill。
- Claude 官方文档确认仓库 Skill 使用 `.claude/skills/<name>/`，支持自包含脚本、SKILL.md 实时变更检测和目录符号链接；本机 Claude Code 2.1.205 高于符号链接要求的 2.1.203。
- 本仓库 `core.symlinks=true` 且 `CLAUDE.md -> AGENTS.md` 已作为真实 SymbolicLink（符号链接）工作，因此可以用一份 `.agents/skills/joinquant-archive-sync` 源码和一个 `.claude/skills/joinquant-archive-sync` 目录链接支持两端。

- 已确认：代码、清单和小型文本进入普通 Git（版本管理）；压缩原始数据、Parquet 和大日志进入 Git LFS（大文件存储），使远端仓库可完整恢复且不把二进制内容写入普通 Git 历史。
- 已确认：首版新增 DuckDB（嵌入式分析数据库）依赖并提供 SQL（查询语言）；DuckDB 只创建读取 Parquet 的视图，不保存重复数据库副本。
- 已确认：采用混合单包管道。Research API（研究接口）负责结构化运行数据，已登录 Playwright 页面适配器负责对象、代码和日志，两路证据统一进入一个临时包并通过相同清单校验后落盘。
- 已确认：使用 `.agents/skills/joinquant-archive-sync/` 单一真实目录，`.claude/skills/joinquant-archive-sync` 以符号链接指向同一目录；SKILL.md、全部脚本、requirements 和 references 均在该 Skill 内。

## 关键取舍与风险

- 普通 Git 保存大量二进制和压缩文件会永久放大历史。
- Git LFS 需要远端支持与容量配额，但能保持仓库可克隆恢复。
- 忽略大数据最省 Git 空间，但新环境无法只靠仓库恢复完整归档。
- DuckDB 提供更直接的 SQL 查询，但增加一个运行依赖；用户已接受该取舍。
- 浏览器直取无法绕开已验证的持仓分页上限；导出会话给本地客户端会扩大 Cookie（浏览器凭证）暴露面。
- 混合管道比纯浏览器多一个 Research 导出步骤，但这是已验证能完整取得结构化结果的最小可靠方案。
- 仓库 Skill 方案删除双市场、双 manifest、安装缓存、版本同步和更新命令；代价是只服务当前仓库，正好符合用户范围。
- Skill 的 SKILL.md 变更由两端自动检测；脚本每次都从工作树执行，因此脚本更新无需缓存刷新。首次创建顶层 Skill 目录后，已打开的会话可能需要一次重启才能开始监听。

## 测试策略

- 先用一个真实回测执行一次性 PoC（概念验证），验证专用登录、Research API、官方导出、Playwright 自动下载和本地落盘；自动下载失败时验证人工导入同一 staging（暂存）协议，两条路径都失败则停止实现并报告阻断。PoC 不进入常规回归。
- 常规 E2E（端到端）完全使用内存模拟：正式 Skill/CLI 提供最小 `self-test` 子命令，在进程内生成小型证据并复用生产同步核心，完整执行导入、校验、临时归档、幂等增量、DuckDB 查询和 CSV 导出；DuckDB 使用内存库，归档只写系统临时目录，不访问网络或历史数据。
- Codex 从 `.agents/skills`、Claude 从 `.claude/skills` 分别调用同一 `self-test`；两端 `SKILL.md` 和脚本 SHA256 必须一致。Windows Task Scheduler 通过 `schtasks /Run` 调用同一 `self-test`。
- 最小回归集覆盖明确目标校验、分页结束、普通日志 1000 条边界与下一页探测、逐数据集完整性门禁、异常 JSON 容错、增量 cursor（游标）和原子提交。
- 内存证据包含最小完整归因日志序列，并覆盖缺页、断序、无终止事件和源站无写入器；同时覆盖完成、失败、取消、免费封顶、源站缺失和 API 版本不支持，防止局部完整被误报为整体完整。
- 交付只走仓库 PR（拉取请求），不创建插件、市场清单、外部发布或插件缓存。Skill 与脚本随仓库合并即为新版本；回滚只回退 Skill 代码，不改写既有归档。

## Spec Patch

- 在“普通日志达到免费边界”场景中补充精确行为：必须获取前 1000 条并执行第 1000 条后的边界探测；少于 1000 条且明确结束不得标记例外。
- 替换“Plugin 和 Skill 必须复用同一同步核心”为“仓库 Skill 必须提供统一同步入口”：一份 `.agents/skills/joinquant-archive-sync` 源目录自包含 SKILL.md、脚本、依赖和参考；`.claude/skills` 使用同仓库符号链接复用该目录；计划任务调用 Skill 内同一脚本。
- 移除双 marketplace、双 plugin manifest 和插件更新契约；改为验收 Codex/Claude 双端发现、同一源码摘要和工作树脚本下一次调用生效。
- 把归因日志补充为独立核心数据集：存在写入器时必须完整取得并严格校验，任何缺页、断序或缺少终止证据都不得通过门禁。
- 明确性能边界：常规端到端回归全部使用运行时生成的内存证据，不访问网络或历史数据；归因日志用最小完整序列验证相同门禁。真实在线验证只作为首次 PoC，不属于端到端回归。
