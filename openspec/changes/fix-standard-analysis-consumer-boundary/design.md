## Context

`package_source`（结果包读取器）先进入遗留 `analysis_data`（分析数据）来源分支，分析计划以仓库根目录解析基线配置，且运行前没有核对计划与结果包的完整场景集合。这些都是既有标准结果包流程的实现偏差。

## Goals / Non-Goals

**Goals:**

- 只通过既有标准结果包校验器读取共同事实，不让生产者类型进入分析分支。
- 让分析计划及其声明的基线配置可整体移动；仓库目录只保留为本地交付位置。
- 在交付前拒绝缺少计划场景或多余场景的结果包集合。

**Non-Goals:**

- 不修改活动 OpenSpec（开放规格）、不新增 Skill（技能）或命令参数。
- 不兼容读取未转换的 JoinQuant（聚宽）归档，也不改动上游研究、归档或云端流程。

## Decisions

1. `package_source` 直接使用现有标准结果包校验器并按该契约的固定路径读取数据，移除对遗留 `AnalysisSource`（分析来源）及其 `kind`（来源类型）的依赖。这样不新增适配器或第二套契约。
2. 将 `baseline_config`（基线配置）相对分析计划文件解析，并继续把其摘要纳入分析身份；`--repository` 只决定 `.local`（本地输出）交付位置。
3. 在已有“重复和未知场景拒绝”检查旁加入集合相等检查。需要部分分析时，由调用方提供只含目标场景的分析计划，不增加选择参数。

## Risks / Trade-offs

- [未转换的遗留归档不再可被分析读取器直接接收] → 这些对象本就不满足标准结果包契约；在生产或归档边界完成转换。
- [旧计划把基线配置放在仓库内] → 保持相对路径写法，现有计划无需迁移。
- [完整场景要求会拒绝不完整批次] → 失败信息明确指出缺失和多余场景，调用方可提供缩小后的计划。
