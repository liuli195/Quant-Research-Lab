## Context

仓库目前只有策略索引骨架，正式回测和模拟交易均在聚宽云端运行。本地系统需要处理四类事实：页面对象与代码、Research API（研究接口）结构化结果、页面日志、以及研究环境产生的归因日志。实测表明结构化回测数据可完整分页取得，但普通日志存在免费上限，部分日志和性能响应包含不合法 JSON（数据格式），传输标识还会随页面刷新变化，因此不能用“请求成功”或单个远端标识代表完整。

同步既包含由用户或 Agent（代理）明确触发的历史回测，也包含北京时间 04:00 的活动模拟交易计划任务。计划任务不能依赖正在运行的 Codex 会话。

## Goals / Non-Goals

**Goals:**

- 使仓库目录与聚宽策略、回测、模拟交易页面一一对应，并保留完整代码、数据、报告和可获得日志。
- 用可重复验证的清单和摘要证明每个数据集是完整、受免费上限限制、源端缺失或接口不支持。
- 支持目标驱动的历史同步、活动模拟交易增量同步、紧凑查询和按需 CSV（表格文件）交付。
- 让 Codex（代码代理）、Claude（代码代理）、计划任务和人工调用通过同一个仓库 Skill 复用同一套脚本及校验逻辑。

**Non-Goals:**

- 不在本地运行正式回测或模拟交易。
- 不默认扫描或下载全部历史运行，不提供 `latest`（最新）选择器，不默认消耗积分。
- 不恢复源代码未产生的归因日志，不修改 Vibe-Trading（AI 交易研究工具）核心。
- 不提交登录凭证，不引入常驻进程或第二套同步实现。

## Decisions

### 1. 页面对象目录是唯一归档边界

使用以下稳定结构；页面刷新后变化的传输标识仅写入 `manifest.json`（清单）别名，不作为目录主键：

```text
joinquant/strategies/<strategy_id>/
├── default_code.py
├── manifest.json
├── backtests/<backtest_id>/
│   ├── code.py
│   ├── params.json
│   ├── manifest.json
│   ├── raw/
│   ├── data/
│   └── reports/
└── simulations/<simulation_id>/
    ├── source.json
    ├── current_code.py
    ├── code_versions/
    ├── params.json
    ├── snapshots/
    ├── manifest.json
    └── reports/
```

`strategy_id` 使用本地稳定键，历史回测使用策略页面内可复核的序号或稳定页面键；所有远端标识和详情链接作为别名保存。选择中央数据仓库再用指针回链虽然便于统一查询，但会破坏页面对应关系并增加引用失效风险，因此不采用。

聚宽页面和官方 `get_backtest` 不存在独立构建对象，因此不创建 `builds/` 或 `sync-build`。

### 2. 原始证据与查询数据各保留一份

- 页面和接口原始响应保存为 gzip（压缩）后的 JSON/JSONL，并记录 SHA256（完整性摘要）。无法解析的响应也必须原样保存。
- 结构化事实表保存为 Parquet + Zstd（列式压缩格式），按对象和数据集拆分；不提交重复的 DuckDB（嵌入式分析数据库）数据库文件。
- DuckDB 查询入口只生成指向 Parquet 文件的视图；Vibe-Trading 仅在请求时物化所需 CSV。
- 临时下载和未验证文件进入 `.local/`，只有通过校验后才原子移动到页面对象目录。

只保存 Parquet 体积更小，但无法证明解析器未丢失源字段；只保存原始 JSON 查询成本高。因此保留一份压缩证据和一份列式查询数据是满足完整性与可用性的最小组合。压缩原始数据、JSONL 日志和 Parquet 使用 Git LFS（大文件存储）；代码、清单和小型文本仍使用普通 Git。实施前先验证当前远端仓库支持 LFS，避免生成无法恢复的归档引用。

### 3. 一个 Python 同步核心，入口保持薄

仓库内只实现一套 Python CLI（命令行接口），提供认证、列出候选目标、同步指定回测、同步活动模拟交易、校验、查询、导出 CSV 和 `self-test` 等命令。唯一真实 Skill 位于 `.agents/skills/joinquant-archive-sync/`，自包含 `SKILL.md`、脚本、依赖与参考资料；`.claude/skills/joinquant-archive-sync` 使用相对 SymbolicLink（符号链接）指向该目录。人工命令和 Windows Task Scheduler（Windows 任务计划程序）调用 Skill 内同一 CLI，不复制抓取或校验逻辑。

复用现有 `.venv`、Playwright（浏览器自动化）、Pandas（数据处理）和 PyArrow（列式数据处理）。不增加 Selenium（浏览器自动化框架）、数据库服务或常驻调度器。

### 4. 浏览器只承担认证和页面能力

- 首次认证命令打开 Playwright 专用持久浏览器目录，由用户登录一次；该目录位于仓库外。
- 实测 Chromium（浏览器）正常关闭后不会跨 CLI（命令行接口）进程保留聚宽的会话 Cookie（浏览器凭证）。因此认证成功后只筛选聚宽域名 Cookie，在内存中交给 Windows DPAPI（数据保护接口）加密，密文保存在同一仓库外目录，仅当前 Windows 用户可解密；不保存账号、密码、明文 Cookie、`storage_state`（浏览器状态导出）或非聚宽域名 Cookie。
- 后续交互式和计划任务先恢复浏览器目录，再恢复 DPAPI 密文中的聚宽会话。密文损坏、换用户或登录失效时返回 `auth_required`，不自动填写密码，也不降级为明文存储。
- 页面结构、代码和日志通过已登录页面或同源请求取得；结构化回测数据优先使用已验证的 Research API。
- 官方文件导出由 Playwright 捕获下载事件保存到临时目录，避免把大文件通过 Agent 对话传输。

Codex 内置浏览器适合调查和人工核验，但不能作为凌晨计划任务运行时。明文 Cookie 文件会扩大凭证暴露面，因此不采用；DPAPI 密文只补足实测缺失的跨进程会话持久化能力。

### 5. 同步采用“前后清单围栏”和原子提交

一次同步先读取目标清单，再逐数据集分页至明确终止条件，最后重读目标清单。校验至少覆盖对象状态、行数、字段、唯一键、排序、时间范围、页终止状态、代码摘要和文件摘要。前后清单不一致时只重试变化部分，不提交半成品。

每个数据集必须落入明确状态：

- `complete`：通过全部完整性检查。
- `capped_free`：普通日志达到免费边界，当前免费数据完整但不能证明全量。
- `missing_at_source`：源代码未产生该类日志或数据。
- `unsupported_api_version`：聚宽明确拒绝当前接口版本。
- `failed`：网络、认证、解析或完整性校验失败。

仅记录最后同步时间不能发现同一时间戳下的数据变化，因此不采用；摘要和游标共同决定增量范围。

### 6. 日志按分析价值和成本分层

- 归因日志是独立核心数据集，默认全量同步。回测只读取目标源码明确指定的单一文件，除 Token（标识）、序列和运行边界外，还必须把 `run_end` 的 `total_value`、`cash` 与 Research 最终资金记录关联。模拟交易仍从全部代码历史的 `sourceBacktestId` 取回真实源码并保存“历史顺序/时间 → 源 ID → 代码 SHA256/文件”映射，但归因所有权取自模拟交易启动时实际初始化的代码版本，只读取该版本的单一路径；后续代码版本中仅出现的回测或实验路径不得下载。归属不唯一、启动源码缺失、运行边界或回测最终资产关联不符时对象门禁失败；启动代码证据证明无写入器时才允许 `missing_at_source`。
- 失败运行错误日志默认完整同步。
- 普通控制台日志和性能响应默认取得全部免费范围并至少覆盖前 1000 条。少于 1000 条且明确结束时标记 `complete`；到达 1000 条后必须探测下一页，免费接口仍可继续时继续下载，只有后续存在但不可免费取得或无法证明结束时才标记 `capped_free`。
- 积分导出必须显式指定运行和日志类型，执行前展示预计范围与聚宽返回的积分价格；未确认不得发起。
- 对不合法 JSON 先保存原始响应，再使用容错解析器恢复可识别记录并报告恢复数量。

将全部日志默认积分下载能提高覆盖率，但产生不可控费用且大部分普通日志不影响常规归因，因此不采用。

### 7. 两种调度路径共用同一同步命令

- 历史回测：调用者必须传入一个或多个明确页面序号或详情链接；缺失、冲突或 `latest` 参数直接拒绝。候选列表只读取轻量元数据，不触发同步。
- 模拟交易：Windows Task Scheduler 每天北京时间 04:00 调用“同步全部活动模拟交易”；失败后每 30 分钟重试，最多 3 次。运行关闭后执行一次最终同步并停止跟踪，次日自动补齐前次失败的增量。

不使用自建守护进程；Windows 原生任务计划已经覆盖单机定时和重试需求。

### 8. 单一 change 和单一能力规格

页面层级、同步、存储、日志、查询及仓库 Skill 共用同一 manifest（清单）和完整性门禁。任一部分缺失都会使端到端同步不可验收，因此保持一个 change（变更）和一个 `joinquant-archive-sync` 能力规格。

## Risks / Trade-offs

- [聚宽页面或内部接口变化] → 将页面访问集中在单一适配层，保存原始响应并以端到端回归发现变化。
- [Playwright 自动下载尚未在专用登录会话完成真实落盘验证] → 将该链路作为第一项 PoC（可行性原型）；失败时保留官方手工下载文件加显式导入路径，不宣称全自动。
- [登录状态过期导致凌晨任务失败] → 返回 `auth_required`、保留失败记录并按既定次数重试；重新登录后从摘要和游标续传。
- [免费日志存在未知上限] → 使用 `capped_free` 与 `complete` 严格区分；只有显式授权才预览并购买指定日志。
- [原始证据和 Parquet 占用双份空间] → 两者都压缩且按对象拆分，并通过 Git LFS 保存大数据；不预建分片服务或远端对象存储。
- [内存端到端回归不证明真实浏览器链路] → 首次实施单独执行一次真实 PoC；PoC 失败时先验证人工导入兜底，两条路径均失败则停止实施。
- [计划任务重复执行] → 使用对象级文件锁和临时目录原子提交；首版只支持单机单用户并发。

## Migration Plan

1. 先用一个已完成回测验证专用登录、官方导出、本地落盘和完整性检查的端到端 PoC。
2. 建立最小 CLI、页面目录、manifest 和原始/Parquet 转换；对同一回测重复同步验证幂等性。
3. 增加归因日志、错误日志、免费日志边界和源端缺失状态。
4. 增加活动模拟交易增量同步及 Windows Task Scheduler 安装/卸载命令。
5. 增加 DuckDB 视图、按需 CSV、仓库 Skill、Claude 符号链接和全内存端到端回归。

回滚时停用计划任务并移除同步入口；已通过摘要校验的归档文件保持只读，不删除用户数据。失败批次仅清理其 `.local/` 临时目录。

## Open Questions

- 专用 Playwright 登录会话能否稳定完成聚宽真实导出并捕获到本地路径，必须由首个 PoC 实测确认。
- 积分日志价格只能在指定导出请求前由聚宽返回，不能预先固定。
- Git LFS 远端支持和配额必须在提交第一批归档前验证；首次真实落盘继续记录压缩比和容量趋势。
