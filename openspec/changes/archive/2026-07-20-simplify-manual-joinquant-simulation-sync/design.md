## Context

现有活动模拟交易归档同时包含手动 `sync-active-simulations`（活动模拟交易同步）入口和每日 04:00 的计划任务发布闭环。后者依赖专用 worktree（工作树）、运行锁、自动分支、PR Flow（拉取请求流程）和 Windows 任务计划程序；其恢复状态会阻塞后续自动批次。

## Goals / Non-Goals

**Goals:**

- 只保留显式调用的 `sync-active-simulations --repository .` 手动同步。
- 同步继续复用现有采集、归档清单和完整性门禁，但不创建工作树、提交、分支、PR 或计划任务。
- 删除自动化编排、调度 CLI（命令行接口）和对应测试、文档要求。

**Non-Goals:**

- 不改变聚宽采集、游标、清单、文件摘要、查询、导出或认证语义。
- 不新增手动发布命令、Git 自动提交、PR 自动化、通知服务或替代调度器。
- 不在代码变更中自动删除现有 Windows 任务或运行时目录。

## Decisions

### 1. 手动同步沿用已有入口

保留 `sync-active-simulations`，以调用者指定的当前仓库为写入位置。它只产生经现有门禁验证的归档文件和结果，不调用 Git 或 PR Flow。重新实现第二套手动同步器会复制采集和清单逻辑，因此不采用。

### 2. 完整移除自动编排，而非保留兼容开关

删除 `scheduled-sync-pr`、`schedule-install`、`schedule-status`、`schedule-uninstall` 及其工作树、锁和运行状态实现；不保留“关闭自动化”的配置。用户已在 PR（拉取请求）前单独授权卸载旧任务，并移除专用工作树，同时保留遗留分支和提交作为恢复证据。

### 3. Git 发布改为用户显式决定

手动同步完成后，归档文件保留在当前工作区；是否检查差异、提交、进入 PR Flow 或舍弃修改均由用户另行决定。这样不再需要自动分支恢复、变基或远端 PR 状态机。

### 4. 仅保留直接入口验证

删除计划任务 XML（配置文本）与自动发布回归；保留 `self-test`（自检）和从 `sync-active-simulations` 手动入口进入的受控端到端回归。现有核心归档校验测试继续覆盖数据完整性。

## Risks / Trade-offs

- [旧任务在代码部署后仍被触发] → 用户已单独授权卸载 `JoinQuantArchiveSync`；不由程序自行删除任务。
- [手动同步后工作区出现归档修改] → 这是显式手动操作的预期结果；工具不暂存、提交或清理，用户自行决定版本管理动作。
- [旧专用工作树占用磁盘] → 停用后不再使用；是否删除运行时目录留待用户单独授权，避免误删可恢复证据。

## Migration Plan

1. 用户单独确认后卸载 `JoinQuantArchiveSync` 任务并移除专用工作树；保留遗留分支和提交。
2. 合并删除自动化能力的代码和文档变更。
3. 用户需要归档时，显式运行 `sync-active-simulations --repository .`，再按普通 Git（版本管理）和 PR Flow（拉取请求流程）流程处理产生的修改。
4. 回滚只恢复已合并的代码版本；不自动重新安装计划任务。
