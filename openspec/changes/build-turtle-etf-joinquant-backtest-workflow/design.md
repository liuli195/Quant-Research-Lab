## Context

仓库已有 `joinquant-archive-sync`（聚宽归档同步），负责认证、明确目标选择、回测同步和归档完整性门禁，但尚无能力把一组正式回测按固定身份、清单顺序和覆盖要求编排起来。海龟 ETF（交易型开放式指数基金）方案又包含功能校验、统一基线和大量会改变交易路径的稳定性场景；若依赖人工表格和“最近一次回测”，容易发生代码、参数、区间或结果错配。

本设计受三项边界约束：正式交易路径只由 JoinQuant（聚宽）云端产生；通用 Skill（技能）不得依赖海龟规则；认证和归档继续复用现有能力。主要使用者是研究人员、Codex（代码代理）和 Claude（代码代理），下游使用者是本地稳健性分析流程。

## Goals / Non-Goals

**Goals:**

- 建立配置驱动、与策略解耦的 `run-joinquant-backtests` Skill（聚宽回测编排技能）。
- 用确定性脚本完成清单校验、摘要、去重、顺序执行、恢复和覆盖率判断。
- 在通用能力之上表达海龟 ETF 的功能校验、正式基线、比较对象和稳定性场景矩阵。
- 每个正式场景都关联唯一代码、参数、区间、费用、远端结果和通过门禁的本地归档。

**Non-Goals:**

- 不在本地运行或重建正式回测，不形成最终稳健性结论。
- 不把海龟规则、参数、策略 ID（标识符）或场景列表写入通用 Skill 和脚本。
- 不实现认证、抓取、归档或查询的第二套逻辑，不修改 `joinquant-archive-sync` 的契约。
- 不启动前向模拟，不修改 `strategy-001` 和 `strategy-002` 的策略规则。

## Decisions

### 1. 三层结构和单向依赖

`SKILL.md` 只描述编排步骤和停止状态；通用 Python（编程语言）入口承担确定性计算；项目目录保存策略代码、场景清单、运行状态和结果引用。通用层只读取调用方传入的 schema（结构约束），不得导入海龟代码或读取海龟方案文档。

替代方案是把海龟矩阵写入 Skill，或让 Skill 内嵌大量计算逻辑。前者无法复用，后者难以测试和从其他入口调用，因此不采用。

### 2. 场景清单是计划事实源，运行台账是状态事实源

版本化清单显式记录策略身份、代码路径及 SHA-256（文件摘要）、参数、日期、资金、费用、执行口径和期望证据。功能校验、正式基线、比较对象和稳定性场景使用固定阶段，各阶段中的场景按清单顺序执行，不开放自定义依赖图。脚本对规范化内容生成场景指纹；同名不同指纹立即失败，同指纹只保留一个执行节点。运行台账只记录远端提交、明确详情链接、归档状态和错误，不反向修改清单。

替代方案是从远端历史记录推断配置，或把计划和状态混在同一可变文件中。两者都无法可靠识别漂移和恢复，因此不采用。

### 3. 远端执行与本地确定性计算隔离

本地脚本按清单顺序检查前置条件并记录结果；Skill 按清单通过获准的聚宽交互入口提交或恢复任务。只有聚宽返回的运行身份和结果才能进入台账，本地测试数据必须标记为测试证据，不能升级为正式结果。未授权、认证失效或需要人工选择时返回明确停止状态，不猜测目标。

替代方案是逆向实现一套未经确认的远端 API（接口），或完全依赖人工顺序。前者扩大安全和维护风险，后者失去可重复性，因此采用受控交互入口加确定性台账。

### 4. 归档通过公开入口交接

每个完成场景把明确回测详情链接交给 `joinquant-archive-sync`，并读取其 manifest（清单）门禁结果；不导入其内部模块，也不复制认证和归档代码。`latest`、`all`、空目标和仅有裸远端 ID 均不能作为归档目标。

替代方案是直接依赖归档器内部实现。公开 CLI（命令行接口）边界更稳定，也能保留现有安全约束，因此不采用内部耦合。

### 5. 海龟矩阵由冻结基线加单项覆盖生成

海龟项目清单先定义一份基线，再分别生成参数、时期、滚动窗口、资产删除、费用、滑点和延迟执行场景。除目标维度外，其余设置继承基线并写入展开后的场景记录；滚动窗口由固定算法生成并锁定数据截止日。这样既避免手写漂移，也不会把海龟内容带入通用脚本。

替代方案是逐场景复制完整配置，或运行后再挑选场景。前者容易漂移，后者违反预设检验要求，因此不采用。

### 6. 覆盖门禁和验证分层

总体完成状态由“期望场景集合”与“通过归档门禁的场景集合”精确比较产生。验证包括通用单元测试、从 `.agents` 真实 Skill（技能）入口运行一次离线 E2E（端到端）自检、Skill Creator（技能创建器）的 `quick_validate.py`、仓库布局测试和仓库完整验证；布局测试单独证明 `.agents` 与 `.claude` 指向同一实现。

替代方案是只检查远端任务显示成功。它不能证明代码对应、归档完整或通用性，因此不采用。

## Risks / Trade-offs

- [Risk（风险）：聚宽页面或交互入口变化] → 将远端动作限制在 Skill 编排层；确定性核心和清单契约不依赖页面结构，失败时保留台账并返回人工处理状态。
- [Risk（风险）：场景数量大、运行中断] → 使用顺序清单、场景指纹和追加式状态记录，仅重试未完成场景，不覆盖已通过门禁的结果。
- [Risk（风险）：远端代码与本地代码不一致] → 功能校验和每个正式场景都比较代码摘要；不一致即停止该场景。
- [Risk（风险）：归档器返回部分证据] → 沿用既有门禁语义，标记证据不完整，不把场景计入覆盖完成集合。
- [Trade-off（取舍）：配置和台账增加前置工作] → 换取可恢复、可审计且不会误用“最近一次回测”的执行过程。

## Migration Plan

1. 新增通用 Skill、确定性脚本、schema、单元测试和离线 E2E 自检，不改变现有归档流程。
2. 通过 `quick_validate.py`、布局测试、一次真实 Skill 入口 E2E 和仓库完整验证。
3. 在海龟项目目录外部于 Skill 的位置创建项目清单；确认策略身份、正式区间、数据截止日和指数来源后执行预检。
4. 先运行聚宽功能校验；通过后分批运行基线和稳定性场景，并逐个使用明确详情链接归档。
5. 覆盖门禁全部通过后，向稳健性分析流程交付场景—归档映射。

回滚时停用新 Skill 和场景清单，不删除既有聚宽任务或已归档证据；恢复前一份台账即可继续。任何远端删除或覆盖均不属于本变更。
