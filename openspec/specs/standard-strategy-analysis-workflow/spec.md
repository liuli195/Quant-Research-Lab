# standard-strategy-analysis-workflow Specification

## Purpose

定义量化分析如何以生产者和存储位置无关的方式，只读消费显式标准结果包、分析计划与独立基准，并产出可重复、可追溯的收益、风险、归因和稳健性结论。

## Requirements

### Requirement: 分析输入必须与结果包位置和生产者解耦

系统 MUST（必须）只要求调用方显式提供一个或多个标准结果包、一个分析计划和一个独立基准清单。结果包的生产者、父目录、文件名和传入路径 MUST NOT（不得）参与场景判断、计算分支或分析身份。

#### Scenario: 同一结果包被移动或复制

- **WHEN** 内容相同的结果包从不同路径传入
- **THEN** 系统产生相同的分析身份和结论

#### Scenario: 输入未显式给出

- **WHEN** 调用方未提供结果包、分析计划或基准清单
- **THEN** 系统停止并报告缺失项，不扫描目录或猜测 `latest`（最新）对象

### Requirement: 结果包必须自证身份和完整性

系统 MUST（必须）从包内清单读取策略、场景和内容摘要，从冻结场景配置读取参数，并验证完成门禁和文件摘要。包内参数 MUST（必须）与分析计划中同一场景的展开参数完全一致；重复场景、重复内容或身份不一致 MUST（必须）被拒绝。

#### Scenario: 包内场景与计划不一致

- **WHEN** 结果包的冻结参数与分析计划不一致
- **THEN** 系统拒绝该结果包，不使用计划参数覆盖真实运行参数

### Requirement: 共同分析和扩展能力必须由数据契约驱动

系统 MUST（必须）使用 `results`（收益）、`balances`（资产）、`positions`（持仓）和 `orders`（订单）共同事实运行统一计算。扩展能力 MUST（必须）按声明字段识别；归因扩展的最小字段为 `time`（时间）、`event_id`（事件标识）和 `event_type`（事件类型），不得依赖扩展名称、执行引擎或生产者类型。

#### Scenario: 海龟归因使用自定义扩展名称

- **WHEN** 任意名称的完整扩展含归因最小字段且摘要、事件身份和时间范围有效
- **THEN** 系统将其作为归因证据，并原样保留已声明的详细字段

#### Scenario: 能力缺失

- **WHEN** 结果包缺少某项分析所需扩展
- **THEN** 系统将该项标记为 `evidence_insufficient`（证据不足），不阻断无依赖的共同分析

### Requirement: 分析交付必须确定、可追溯且只读

系统 MUST（必须）把结果包内容摘要、分析计划摘要、基准摘要、脚本版本和证据矩阵写入确定性 JSON（结构化数据）及 Markdown（标记文档）报告。分析前后输入身份不一致时 MUST（必须）停止。分析 MUST NOT（不得）修改结果包、启动上游流程、联网或读取凭证。

#### Scenario: 离线完整分析

- **WHEN** 调用者从唯一 Skill（技能）入口提供有效结果包、计划和基准
- **THEN** 系统离线完成分析与报告，且结果包字节保持不变
